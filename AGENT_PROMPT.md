3. **Make the change** — implement the step as described. Keep changes minimal; do not fix adjacent issues, even if tempting.
4. **Run verification — pick the narrowest check that proves the change.** `check.sh` runs the full feature matrix and takes 10–20 min; do not invoke it as a default. Instead, scope verification to what the change can actually break:

   | Kind of change | Verification |
   |---|---|
   | Pure docs / comments / Markdown | No build. Just verify no broken local links. |
   | Single crate, no cross-crate signature change | `cargo clippy -p <crate> --all-targets --no-default-features --features multithreading -- -D warnings` and `cargo test -p <crate> --no-default-features --features multithreading -- --nocapture` |
   | Cross-crate refactor that does not touch feature-gated code | Workspace clippy + tests on the default (multithreading) feature set: `cargo clippy --workspace --all-targets --no-default-features --features multithreading -- -D warnings && cargo test --workspace --no-default-features --features multithreading -- --nocapture` |
   | `cfg(feature = ...)` code, default-feature wiring, or anything in `Cargo.toml` `[features]` | Clippy + tests on each feature combination you can plausibly affect (use `cargo run -p xtask -- matrix test-features --format lines` to enumerate). Don't reach for full `check.sh` until the targeted runs pass. |
   | `build.rs`, `crates/dotnet-vm/build_support/`, intrinsic/instruction codegen, or anything env-var driven at build time | `bash scripts/check_build_script_regressions.sh` plus single-feature build of `dotnet-vm`. |
   | Lock-order, GC coordinator, threading, safepoints, or STW | The two lock-order tests in `check.sh` (`stress_lock_order_gc_cycle_guard_with_live_safepoints`, `lock_order_request_stop_the_world_rejects_inverted_top_level_order`) under `--features multithreading`. Run with `--test-threads=1`. |
   | `dotnet-cli` integration / hang-probe behavior | The four hang-probe tests in `check.sh` (`test_allocation_pressure`, `test_gc_coordinator`, `test_multiple_arenas`, `test_stw_stress`) under `--features multithreading`. |
   | Documentation drift checks | `bash scripts/check_doc_drift.sh` |
   | End of a multi-step phase, or any change you cannot confidently scope | `bash check.sh` once, as the phase gate — not per step. |

   Rules of thumb:
   - **Touching `dotnet-vm-ops` or `dotnet-vm` is not, by itself, a reason to run `check.sh`.** A trait rename or pure-Rust refactor is covered by workspace clippy + tests on the default feature set.
   - **Don't pre-emptively run the full matrix "to be safe."** If targeted checks pass and the change is feature-flag-agnostic, that is sufficient signal. The matrix is for catching feature-combination regressions specifically.
   - **If you find yourself thinking "this is risky, I should run everything,"** prefer narrowing the change instead, or call out the risk in your memory entry and let the phase-gate run catch it.
   - Record in your memory entry **which** verification you ran and **why** that scope was sufficient. If you skipped `check.sh`, briefly justify it.
